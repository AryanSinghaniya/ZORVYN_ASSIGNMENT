# Finance Data Processing and Access Control Backend

## Overview
This project is a backend API for managing finance records with secure authentication and role-based access control.

It is designed with an interview-friendly MVC structure and clean module separation so features are easy to explain, maintain, and extend.

## Features
- JWT-based authentication (register and login)
- User management with role and status control
- Roles: admin, analyst, viewer
- User status handling: active, inactive
- Financial record CRUD with soft delete support
- Dashboard analytics APIs using MongoDB aggregation
- Input validation using express-validator
- Centralized not found and global error handling
- Consistent JSON API response format

## Tech Stack
- Node.js
- Express.js
- MongoDB
- Mongoose

## Setup Instructions
1. Clone the repository.
2. Install dependencies.

~~~bash
npm install
~~~

3. Create a local environment file from the example.
4. Update environment values.
5. Start the development server.

~~~bash
npm run dev
~~~

6. Optional: seed sample users and records.

~~~bash
npm run seed
~~~

## Environment Variables
Use the following variables in your .env file:

~~~env
PORT=5000
MONGO_URI=your_mongodb_connection_string
JWT_SECRET=your_secure_jwt_secret
JWT_EXPIRES_IN=7d
NODE_ENV=development
CORS_ORIGIN=http://localhost:3000
~~~

## API Endpoints

### Auth
- POST /api/auth/register
- POST /api/auth/login
- GET /api/auth/me

### Users (Admin only)
- POST /api/users
- GET /api/users
- PATCH /api/users/:id
- GET /api/users/:id
- PATCH /api/users/:id/role
- PATCH /api/users/:id/status

### Records
- POST /api/records
- GET /api/records
- GET /api/records/:id
- PUT /api/records/:id
- PATCH /api/records/:id
- DELETE /api/records/:id

### Dashboard
- GET /api/dashboard/summary
- GET /api/dashboard/trends?period=monthly|weekly
- GET /api/dashboard/category-totals
- GET /api/dashboard/recent-activity
- GET /api/dashboard/monthly-trends

## Roles and Permissions

### Admin
- Full access to all user, record, and dashboard endpoints
- Can create/update/delete records
- Can update user role and status

### Analyst
- Read access for records and dashboard endpoints
- No access to user management
- No create/update/delete record permissions

### Viewer
- Read access for records and dashboard endpoints
- No access to user management
- No create/update/delete record permissions

## Assumptions Made
- Authentication token is sent using Authorization header in Bearer format.
- Soft delete is used for records via isDeleted flag.
- Dashboard calculations ignore soft-deleted records.
- Admin is the only role allowed to manage users.
- Validation errors return HTTP 400 with full validation error list.

## Dashboard API Explanation

- GET /api/dashboard/summary
   - Returns totalIncome, totalExpenses, and netBalance.

- GET /api/dashboard/category-totals
   - Aggregates and returns totals grouped by category.

- GET /api/dashboard/recent-activity
   - Returns latest 5 non-deleted financial records.

- GET /api/dashboard/monthly-trends
   - Groups records month-wise and returns income, expense, and net balance trend values.

## 7. Documentation
Clarity of the README, setup process, API explanation, assumptions made, and tradeoffs considered.

### Setup Process Clarity
- Project setup is documented with install, environment configuration, run, and seed commands.
- .env.example is included for quick onboarding.

### API Explanation Clarity
- Endpoints are grouped by module: Auth, Users, Records, and Dashboard.
- Role restrictions and intended access patterns are documented.

### Assumptions Made
- Role-based permissions are enforced through middleware and route-level guards.
- Soft delete is preferred for records to preserve historical data.
- Seed users are intended for local/demo testing only.

### Tradeoffs Considered
- Chosen: simple JWT access token flow to keep implementation interview-friendly.
- Deferred: refresh token rotation and advanced token revocation for complexity control.
- Chosen: explicit controller logic over heavy abstraction to improve readability for reviewers.
